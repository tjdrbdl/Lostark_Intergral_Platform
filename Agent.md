# Agent.md

## 프로젝트

ArkPilot — 로스트아크 원정대 투자 판단 및 지인 레이드 조율 도구

## 역할

- ChatGPT: 제품 기획과 사용자 가설
- Codex: PM, 구현 조율, 저장소 관리
- VSCode Chat: 세부 개발 보조

## 작업 시작 시 문서 읽기

모든 에이전트는 전체 문서를 무조건 읽지 않습니다. 먼저 아래 공통 문서를 순서대로 읽고, 작업 종류에 맞는 문서만 추가로 읽습니다.

### 공통 필수 순서

1. `Agent.md` — 역할, 작업 규칙, 문서 우선순위
2. `docs/PRD.md` — 현재 MVP 범위, 제외 항목, 성공 기준
3. `docs/IA.md` — 해당 기능의 사용자 흐름과 라우트 책임

### 작업별 추가 문서

| 작업 종류 | 추가로 읽을 문서 | 목적 |
|---|---|---|
| API·데이터·오류 처리 | `docs/API_SCHEMA.md` | 응답 형식, 오류 코드, 데이터 출처 계약 확인 |
| 레이드 모집 기능 | `docs/RAID_PARTY_MVP.md` | 초대방, 참여자, 접근 정책과 제외 범위 확인 |
| 현재 Sprint 구현 | `docs/sprints/current/SPRINT7_HANDOFF.md` | 현재 작업 범위와 완료 조건 확인 |
| 현재 Sprint 검증 | `docs/sprints/current/SPRINT7_QA.md` | 검증 결과와 미해결 항목 확인 |
| 과거 결정 조사 | `docs/sprints/archive/*` | 현재 문서만으로 이유를 알 수 없을 때 참고 |

## 문서 충돌 우선순위

문서 내용이 충돌하면 아래 순서를 따릅니다.

1. 사용자의 현재 요청
2. `docs/PRD.md`
3. `docs/RAID_PARTY_MVP.md` 또는 `docs/API_SCHEMA.md`
4. `docs/IA.md`
5. `docs/sprints/current/*`
6. `docs/sprints/archive/*`

과거 Sprint 문서는 현재 MVP 범위를 확장하는 근거로 사용하지 않습니다. 충돌을 발견하면 추측으로 구현하지 말고 현재 기준 문서를 먼저 갱신합니다.

## 핵심 원칙

1. 폴더 구조보다 사용자 시나리오와 데이터 흐름을 먼저 확정합니다.
2. 프론트는 LostArk 외부 API를 직접 알지 않습니다.
3. 백엔드는 외부 응답을 내부 표준 스키마로 정규화합니다.
4. 실제 API, 샘플 데이터, 계산 기반 추천을 화면과 문서에서 구분합니다.
5. 외부 장애, 실제 미존재, 부분 성공을 서로 다른 상태로 처리합니다.
6. 추천은 정답이 아니라 근거와 신뢰도를 가진 후보로 제공합니다.
7. MVP 제외 기능은 구현하지 않고 backlog로 분리합니다.
8. 기존 사용자 변경과 진행 중인 Sprint 변경을 임의로 덮어쓰지 않습니다.

## MVP 핵심 흐름

### 원정대 투자 판단

검색 → 원정대 통합 결과 → 투자 후보와 근거 확인 → 저장 → 재방문

### 지인 레이드 조율

모집방 생성 → 초대 링크 공유 → 캐릭터·가능 시간 제출 → 방장 일정 확정

## MVP 제외 원칙

- 공개 랜덤 매칭
- 실시간 채팅과 음성 채팅
- 푸시 알림과 캘린더 자동 연동
- 자동 파티 조합과 스펙 합격 판정
- 실제 숙제 완료 자동 추적
- 랭킹, 커뮤니티, 지도·도감, 실시간 시세 포털

레이드 모집은 초대 링크 기반의 지인 조율에 한정합니다.

## 문서 유지 규칙

- 제품 결정은 `docs/PRD.md`에 반영합니다.
- 새 라우트나 흐름은 `docs/IA.md`와 함께 갱신합니다.
- API 계약 변경은 코드와 `docs/API_SCHEMA.md`를 같은 작업에서 갱신합니다.
- Sprint 폴더에는 현재 HANDOFF와 QA만 유지합니다.
- 종료된 Sprint는 핵심 리뷰와 최종 QA만 archive에 보관합니다.
- 중간 진행보고, 완료된 일회용 프롬프트, 중복 요약은 삭제합니다.
